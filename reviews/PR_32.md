## Architectural Revision: No Pre-Filtering of LRR Markers for Association — Propagate QC Flags Instead

The core design principle we discussed needs to be applied here: **rigorous QC for estimation stages (RSVD, GRM), no pre-filtering for association, all QC stats propagated to the output TSV for downstream filtering.**

The current PR applies `variant_qc_mask` (call rate + HWE + MAF) to LRR markers *before* association testing via `--no-apply-variant-qc` / `association_marker_qc.apply_variant_qc`. This is the wrong model for LRR-GWAS. LRR is a continuous intensity signal — genotype-derived QC metrics (MAF, HWE, genotype call rate) don't directly determine whether a marker's LRR is informative. Pre-filtering silently drops potentially interesting loci and forces re-runs if threshold preferences change.

### Three-Tier QC Model (Target State)

| Stage | Marker QC | Rationale |
|---|---|---|
| **RSVD Correction** | call rate + HWE + LRR variance + complexity (**rigorous**) | Noisy markers corrupt the SVD decomposition |
| **GRM Construction** | call rate + HWE + MAF + LD pruning (**rigorous**) | Standard genotype QC for kinship estimation |
| **LRR Association** | INTENSITY_ONLY + monomorphic LRR **only**; all upstream QC flags **propagated as output columns** | Run once, filter the output TSV freely |

### Specific Changes Required

#### 1. `cli.py` — Remove the variant QC pre-filter for LRR association markers

The block that builds a `variant_qc_mask` for LRR markers and filters them before `run_association()` must be removed. The **only** hard-fail pre-filters for association markers should be:
- `INTENSITY_ONLY` exclusion (no cluster model → unreliable LRR) ✅ keep this
- Monomorphic LRR exclusion (zero variance → degenerate statistics) ✅ keep this

The GRM path (lines ~1060-1075 on `main`) remains **unchanged** — it correctly uses `require_call_rate=True, require_hwe=True, require_maf=True`.

#### 2. `cli.py` — Remove `--no-apply-variant-qc` CLI flag (lines 452-466)

This flag controls a pre-filter that should no longer exist. Remove the argument definition entirely. The `--variant-qc` flag still controls:
- GRM marker QC (call rate + HWE + MAF) — unchanged
- QC flag propagation to the output TSV — unchanged and now even more important

Update `--variant-qc` help text to clarify that it applies to GRM only, and that QC flags are written to the output TSV for post-hoc filtering of association results.

#### 3. `qc_config.py` — Remove `apply_variant_qc` from `association_marker_qc`

The config section should only retain:
```yaml

  exclude_intensity_only: true
  exclude_monomorphic_lrr: true
  # apply_variant_qc: REMOVED
```

Remove `"apply_variant_qc": True` from `_DEFAULTS["association_marker_qc"]` (line 157) and update the docstring accordingly.

#### 4. Output TSV — Ensure provenance columns cover all tested markers

The existing provenance logic (lines ~1237-1257 on `main`) already appends `all_ancestries_call_rate_pass`, `all_ancestries_hwe_pass`, and `all_ancestries_maf_pass` to the output TSV. This is the key mechanism — it now applies to **all** tested markers (not just survivors of a pre-filter), making post-hoc filtering trivial:

```bash

awk -F'\t' '$NF_call_rate == "True" && $NF_hwe == "True" && $NF_maf == "True"' results.tsv
```

### Summary

The philosophy: **run association once on the maximally inclusive marker set, annotate everything, filter the output freely.** This is how modern GWAS pipelines work (SAIGE, REGENIE, etc.) and it's especially appropriate for LRR where genotype-derived QC metrics are orthogonal to the signal being tested.
